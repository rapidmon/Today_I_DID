이 사용자가 "온보딩"을 요청하면 **브랜드 쇼케이스 슬라이드(콘셉트 소개, 화려한 비주얼)가 아니라 실제 화면을 모킹한 튜토리얼형 가이드**를 원한다.

**Why:** 같은 프로젝트에서 이전 두 번의 온보딩 작업이 모두 "브랜드 쇼케이스" 수준에 그쳐 사용자가 재작업을 명시적으로 요청. 사용자의 정확한 표현: "실제 기능 사용법을 가르치는 튜토리얼형 온보딩".

**How to apply:** 이 프로젝트(Today I Did) 또는 동일 사용자가 온보딩/사용법/튜토리얼 HTML을 요청할 경우 다음 패턴을 따를 것:

1. **반드시 실제 코드부터 정밀 탐색** — 추측 금지. `app/(tabs)/index.tsx`, `android/.../widget/`, `lib/widgetBridge.ts` 등에서 실제 UI 흐름·제스처·제약사항·상수값을 코드로 확인 후 HTML에 반영.
2. **슬라이드 구조** — 좌측: 360x720 폰 프레임에 실제 앱 UI 모킹 + 하이라이트 링(노/시안/마젠타/빨강 펄스) + 제스처 뱃지("탭", "← 스와이프", "길게 눌러 끌기"). 우측: 번호 매긴 단계별 설명(1.→2.→3.) + 코드상 제약/예외 박스.
3. **17개 등 다수 슬라이드** — 사용자가 슬라이드 수 늘리는 것을 선호. 3~6개로 압축하지 말고 기능별로 충실히 분리. 상단에 인덱스 카드(앵커 점프) 필수.
4. **코드와 일치하지 않는 부분 정정** — 사용자가 메시지에서 부정확한 사실(예: "4단계" 투명도, "하드드롭")을 적었더라도 코드 기준으로 수정하고 HTML에 정정 사실을 명시.
5. **없는 기능은 지어내지 말고 명시** — 예: 앱 내 리셋 버튼 없음 → "앱에는 별도의 리셋 버튼이 없습니다. 위젯 GAME OVER 시 NEW로만 가능"이라고 정확히 기술.
